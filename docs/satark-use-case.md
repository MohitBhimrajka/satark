# Satark — Use Case Documentation

> Required by Agent Rule 12 (Developer Guardrails). Documents the Satark operational loop, data flows, and AI evaluation points.

---

## Use Case Summary

**Satark** enables defence personnel to submit suspicious digital content for AI-powered threat analysis. It provides CERT-Army analysts with a triaged, prioritized incident queue and actionable intelligence.

---

## Operational Data Flow

```
[Submitter: Defence Personnel]
    ↓ submits via browser (URL, text, image, audio, video, document)
    
[Ingestion: POST /api/incidents]
    ↓ validates file type + size
    ↓ uploads evidence to GCS (or local dev filesystem)
    ↓ creates Incident record (status: "pending")
    ↓ returns case number + guest_token immediately
    ↓ triggers background task
    
[AI Analysis: Background Task]
    ↓ orchestrator routes to correct analyzer by input_type
    ↓ builds prompt with SYSTEM_PROMPT_BASE + type-specific template
    ↓ calls Gemini 3 Flash with structured output (ThreatAnalysis schema)
    ↓ validates JSON response with Pydantic
    ↓ updates Incident: classification, threat_score, confidence, ai_analysis JSON
    ↓ auto-sets priority: score>=80→critical, 60-79→high, 40-59→medium, <40→low
    ↓ updates status: "analyzing" → "reviewed" (or "analysis_failed")
    ↓ creates AuditLog entry: action="analyzed"
    
[Submitter: Sees Results]
    ↓ polls GET /api/incidents/:id until status != "analyzing"
    ↓ sees threat score gauge, classification, IOCs, mitigation playbook
    ↓ can share case URL (includes guest_token in URL)
    
[Analyst: CERT-Army Operator]
    ↓ logs in (analyst role)
    ↓ sees prioritized case queue in Workbench
    ↓ can filter: critical first, by classification, by date
    ↓ opens case → reviews AI analysis → adds analyst notes
    ↓ changes status: reviewed → escalated | closed
    ↓ assigns to specific analyst
    ↓ generates PDF report (auto-logged)
    ↓ creates AuditLog: action="status_changed" | "note_added" | "report_generated"
    
[Admin]
    ↓ manages analyst accounts (create, assign roles)
    ↓ monitors platform health (total incidents, avg response time)
    ↓ views all audit logs
```

## AI Evaluation Points (Policy Hooks — Satark Equivalent)

Satark doesn't use a Policy Engine — the AI makes decisions directly. These are the evaluation points:

| Lifecycle Stage | AI Evaluation | Output |
|----------------|--------------|--------|
| **On ingestion** | Analyzes submitted content | classification, threat_score, confidence, indicators, mitigation_steps, risk_factors |
| **On priority assignment** | Automatic based on threat_score | critical (≥80), high (60-79), medium (40-59), low (<40) |
| **On quick scan** | `/api/analyze/*` endpoints | ThreatAnalysis (no Incident created, immediate response) |
| **On report generation** | Not AI-driven — ReportLab formats existing analysis | PDF document |

## Case Number Generation

Format: `SAT-YYYY-NNNNN` (e.g., `SAT-2026-00001`)

```sql
-- PostgreSQL sequence for thread-safe case numbers
CREATE SEQUENCE satark_case_seq START 1 INCREMENT 1;

-- In Incident model (on creation):
case_number = f"SAT-{datetime.now().year}-{nextval('satark_case_seq'):05d}"
```

## Guest Access Security Model

1. Submitter POSTs to `/api/incidents` (no auth)
2. Response includes `guest_token` (UUID4) — submitter must save this
3. Case URL format: `/case/SAT-2026-00001?token=<UUID4>`
4. Backend: `GET /api/incidents/:id?token=<UUID4>` → validates token matches `Incident.guest_token`
5. No expiry in v1 (demo). Token only allows viewing their own case.

| A — Incident Type | Donut | `SELECT classification, COUNT(*) FROM incidents GROUP BY classification` |
| B — Severity Heatmap | Calendar | `SELECT DATE(created_at), AVG(threat_score) FROM incidents GROUP BY 1` |

## PDF Report Structure

Each report is a multi-section PDF generated by ReportLab:

```
Page 1 — Cover
  Satark shield logo (centered)
  Case number (large, navy)
  Classification badge (color-coded)
  Generation date

Page 1-2 — Incident Summary
  Input type, submission date, submitter (or "Anonymous")
  AI Analysis: threat score gauge (drawn with ReportLab SVG), confidence bar

Page 2 — Findings
  Summary paragraph
  IOC list (monospace table: type | value)
  Risk factors (bulleted)

Page 3 — Mitigation Playbook
  Numbered steps

Page 3 — Evidence Log
  File table: name | size | MIME type | SHA-256 (truncated) | uploaded at

Page 4 — Audit Trail
  Timeline table: date | action | actor | details

Footer (all pages)
  "Generated by Satark AI Intelligence Platform | SIH 2025 PS 25210"
```

## Async Analysis Polling Pattern

```typescript
// frontend/src/hooks/usePolling.ts
export function useIncidentPolling(incidentId: string, initialStatus: string) {
  const [status, setStatus] = useState(initialStatus);
  const [analysis, setAnalysis] = useState(null);
  
  useEffect(() => {
    if (status !== 'analyzing') return;
    
    const interval = setInterval(async () => {
      const res = await fetch(`/api/incidents/${incidentId}`);
      const data = await res.json();
      setStatus(data.data.status);
      if (data.data.status !== 'analyzing') {
        setAnalysis(data.data.ai_analysis);
        clearInterval(interval);
      }
    }, 2000); // Poll every 2 seconds
    
    return () => clearInterval(interval);
  }, [incidentId, status]);
  
  return { status, analysis };
}
```
